# HTML → 设计稿 转换流程文档

## 总体流程

```
动态生成的 HTML
       │
       ▼
  [Step 1] Puppeteer 渲染 → 提取布局坐标
       │
       ▼
  [Step 2] LLM 解析 HTML → 语义组件树 JSON
       │
       ▼
  [Step 3] Embedding 搜索 → 匹配组件 Key + 变体
       │
       ▼
  [Step 4] 算法组装 → 画板数据 (frame + instances)
```

---

## Step 1：提取布局坐标

### 做什么
用 Puppeteer / Playwright 真实渲染 HTML，通过 JS 调用 `getBoundingClientRect()` 提取页面中各个区域的实际坐标和尺寸。

### 怎么做
1. 启动无头浏览器，加载动态生成的 HTML
2. 等待页面渲染完成（等待网络空闲或特定元素出现）
3. 在浏览器上下文中执行 JS，遍历语义标签或约定的 class 标记，提取每个区域的 `{ x, y, width, height }`
4. 输出一份 `layoutMap`，key 为区域标识，value 为坐标信息

### 输出示例
```json
{
  "navbar": { "x": 0, "y": 0, "width": 375, "height": 56 },
  "search-input": { "x": 16, "y": 72, "width": 343, "height": 40 },
  "card-list": { "x": 0, "y": 128, "width": 375, "height": 600 }
}
```

### 难点
- **动态内容加载时机**：HTML 如果有异步数据，需要等待内容稳定后再提取，否则坐标不准
- **视口尺寸影响布局**：移动端/PC 端布局不同，需要在提取前设置好目标视口（如 375×812）
- **滚动容器**：页面超出屏幕时，坐标是相对于文档还是视口需要统一处理
- **如何标记需要提取的区域**：HTML 生成时需要约定某种标记方式（如特定 data 属性），才能在 Puppeteer 中准确定位

### 待定项
- [ ] HTML 生成时是否可以在组件根元素上加 `data-component` 属性作为标记？这会大幅降低提取难度
- [ ] 目标设备尺寸是固定的（如 375）还是需要支持多端？

## Step 2：LLM 解析 HTML → 语义组件树

### 做什么
将 HTML 结构输入给 LLM，让其识别出页面由哪些语义组件构成，并提取每个组件的关键样式特征，输出结构化 JSON。

### 怎么做
1. 设计固定的 prompt 模板，包含：
   - 任务描述（识别组件、提取特征）
   - 输出 schema 约束
   - 少量 few-shot 示例
2. 将 HTML（或精简后的 HTML 骨架）作为输入
3. LLM 输出标准化的语义组件树 JSON

### 输出示例
```json
{
  "type": "frame",
  "children": [
    {
      "id": "navbar",
      "name": "顶部导航栏",
      "props": {
        "hasBackButton": true,
        "title": "首页",
        "hasRightAction": false
      }
    },
    {
      "id": "search-input",
      "name": "搜索输入框",
      "props": {
        "placeholder": "搜索",
        "hasIcon": true
      }
    }
  ]
}
```

### 难点
- **HTML 太长**：动态生成的 HTML 可能很大，直接塞给 LLM 会超 token 限制。需要先做 HTML 精简（去掉样式、脚本，只保留结构骨架）
- **输出格式稳定性**：LLM 输出可能不严格遵守 schema，需要做输出校验和容错解析
- **组件识别粒度**：LLM 可能把一个大区域识别成多个小组件，或把多个组件合并识别，粒度控制是难点
- **样式特征提取**：颜色、大小等信息需要从 HTML 的 inline style 或 class 推断，需要在 prompt 中明确指导

### 待定项
- [ ] HTML 精简策略：是去掉所有 class/style，还是保留用于变体判断的样式信息？
- [ ] 当 LLM 识别结果置信度低或结构异常时，如何处理？是报错还是降级？
- [ ] 是整页一次性解析，还是按区块分段解析？（影响准确率和 token 消耗）

## Step 3：Embedding 搜索 → 匹配组件 Key + 变体

### 做什么
根据 Step 2 输出的组件名称和样式特征，从云端组件库中搜索最匹配的组件，确定唯一的组件 Key 和具体变体。

### 怎么做

**离线准备（组件库预处理）：**
1. 遍历组件库所有组件及其变体
2. 为每个变体生成描述文本（名称 + 用途 + 特征，如"主要按钮 大尺寸 蓝色背景"）
3. 用 Embedding 模型将描述文本向量化，存入向量数据库

**在线匹配：**
1. 将 Step 2 输出的组件名 + props 拼成查询文本
2. 向量化后在数据库中做相似度搜索，取 top-K 候选
3. 如果 top-1 置信度超过阈值，直接使用
4. 如果置信度不足，将 top-3 候选交给 LLM 做最终选择

### 输出示例
```json
{
  "id": "navbar",
  "componentKey": "abc-123-xyz",
  "variantKey": "default/with-back-button",
  "confidence": 0.92
}
```

### 难点
- **组件库变体描述的质量**：向量搜索的准确率完全依赖描述文本的质量，如果组件库没有结构化的变体描述，需要人工补充或自动生成
- **同类组件区分度低**：比如"输入框"和"搜索输入框"向量相近，容易混淆，需要在描述中强化差异点
- **变体维度多**：一个组件可能有尺寸、颜色、状态等多个维度的变体，需要同时锁定所有维度
- **组件库更新同步**：组件库新增/修改组件后，向量索引需要增量更新

### 待定项
- [ ] 组件库现有的数据结构是什么？变体是如何存储和描述的？
- [ ] 是否有现成的向量数据库可用？还是需要搭建？
- [ ] 置信度阈值如何定？低于阈值时的兜底策略是什么（报错/跳过/用默认组件）？
- [ ] 组件库中完全没有对应组件时怎么处理？

---

## Step 4：算法组装 → 画板数据

### 做什么
将 Step 1 的布局坐标 + Step 3 的组件 Key，组装成最终的画板数据格式（frame + instances）。

### 怎么做
1. 以页面为单位创建 frame 节点（取视口尺寸）
2. 遍历所有匹配到 Key 的组件，查 `layoutMap` 拿坐标
3. 每个组件生成一个 instance 节点，填入 Key 和坐标
4. 按 y 坐标排序，组装成 children 数组
5. 输出最终 JSON

### 输出示例
```json
{
  "type": "frame",
  "x": 0,
  "y": 0,
  "width": 375,
  "height": 812,
  "children": [
    {
      "type": "instance",
      "componentKey": "abc-123-xyz",
      "x": 0,
      "y": 0,
      "width": 375,
      "height": 56
    },
    {
      "type": "instance",
      "componentKey": "def-456-uvw",
      "x": 16,
      "y": 72,
      "width": 343,
      "height": 40
    }
  ]
}
```

### 难点
- **坐标体系对齐**：Puppeteer 拿到的坐标是相对于文档顶部的，需要确认画板坐标系一致
- **部分组件匹配失败时**：Step 3 中有些组件没匹配到 Key，这条数据是跳过还是用占位符填充？
- **组件尺寸覆盖**：instance 的宽高用 Puppeteer 提取的实际尺寸，还是组件库默认尺寸？两者可能不一致

### 待定项
- [ ] 画板数据的最终格式规范（由接入的设计工具决定，如 Figma / 自研工具）
- [ ] 是否需要支持嵌套 frame（如页面内有多个 section）？

## 整体待定项汇总

| # | 问题 | 影响步骤 | 优先级 |
|---|---|---|---|
| 1 | HTML 生成时能否加 `data-component` 标记 | Step 1 | 高 |
| 2 | 目标视口尺寸是否固定 | Step 1 | 高 |
| 3 | 组件库变体的数据结构 | Step 3 | 高 |
| 4 | 向量数据库基础设施是否已有 | Step 3 | 高 |
| 5 | 匹配失败时的兜底策略 | Step 3、4 | 中 |
| 6 | HTML 精简策略（保留哪些样式信息） | Step 2 | 中 |
| 7 | 画板数据最终格式规范 | Step 4 | 中 |
| 8 | 是否支持嵌套 frame | Step 4 | 低 |
